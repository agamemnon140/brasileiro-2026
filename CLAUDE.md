# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A single-page React app that simulates the 2026 Brazilian Brasileirão (Séries A, B, C, D + Copa do Brasil) using an Elo + Poisson goal model run through Monte Carlo. It is a PWA (installable, offline-capable) shipped as **one standalone HTML file** with React/ReactDOM/Tailwind pulled from CDNs. UI text is in Brazilian Portuguese.

## Repo

The working repo is named `brasileirao-2026` (same name as the app and its built HTML); this directory is the project root.

The real source is `simulador_unificado.jsx` — a single ~18k-line React component file. If you are asked to change app behavior, you generally need the `.jsx` source. If only `brasileirao-2026.html` is present, you are looking at compiled output — flag this rather than editing the transpiled JSX by hand. (Exception: when the `.jsx` is unavailable, a small, well-isolated value change can be applied directly to the HTML — e.g. the Anthropic model id was updated in-place to `claude-sonnet-4-6`. Mirror any such change back into the `.jsx` once it's restored, or it will be lost on the next rebuild.)

## Build

```bash
npm install @babel/core @babel/preset-react   # no package.json exists; install ad hoc
node build_html.js
```

`build_html.js` does, in order: strips `import React` and `export default` from the jsx; exposes hooks as globals; appends a `ReactDOM.createRoot(...).render(...)` bootstrap; transpiles JSX with `@babel/preset-react` (classic runtime); injects the PWA manifest + icons from `icon_b64.json`; and writes the final HTML. **Note both paths are hardcoded** — input `/home/claude/simulador_unificado.jsx`, output `/mnt/user-data/outputs/brasileirao-2026.html` — adjust them for a local build.

There is no test suite, linter, or dev server. To run the app, open `brasileirao-2026.html` in a browser (it's fully self-contained).

## Architecture of the app (inside the jsx / transpiled HTML)

Top-level component `SimuladorUnificado` holds a `tab` state switching between: `serieA`/`serieB`/`serieC` (→ `LeagueSim`), `serieD` (→ `SerieDSim`), `copaBR` (→ `CopaBrasilSim`), `h2h` (→ `ConfrontoMerged`), `config` (→ `SettingsTab`). A "Simular Tudo" dashboard is the **single Monte Carlo channel** — individual tabs only keep a "1 Sim rápido" button.

**Simulation engine** (pure functions, no React): `poissonProb` → `calcProbs`/`calcL` (Poisson goal expectations from Elo + atk/def), `updR` (Elo + atk/def update after a game), `applyDrift`/`boxMuller` (per-sim structural-uncertainty noise on Elo and atk/def), `initLeague`/`computeCurrentAD`/`backtestSeries` (derive current team strength from real results played so far), `simMC` (N-run Monte Carlo, 1k or 10k) and `simUnica` (single deterministic-ish run). Each series has special post-season logic: Série B access playoff (3º–6º), Série C final quadrangular (G8 → 2 groups → final ida/volta), Série D's large multi-phase knockout handled via `prepareSerieDState` + `pairKey` (matches results to fixtures by sorted-pair+leg, not exact home/away/round).

**Per-series data** is hardcoded as `S{A,B,C}_TAB` (fixtures), `_RES` (real results played), `_RANKING`, `_DATES`, `_NM`, plus `_META` objects (e.g. `nReb` relegation count, zone cutoffs — these encode 2026 CBF format changes like Série C having only 2 relegated). Copa do Brasil uses `CB_RES_IDA`/`CB_RES_VOLTA`. Results are merged with user-fetched results from `localStorage`.

**`UpdaterPanel`** ingests new real results: it downloads a CBF PDF through the `allorigins.win` CORS proxy, then calls the Anthropic Messages API (`api.anthropic.com/v1/messages`, model `claude-sonnet-4-6`) with the PDF as a document block to extract finished matches as a JSON array, which is merged and persisted. API key resolution order: `window.__ANTHROPIC_API_KEY__` (build-time placeholder `__ANTHROPIC_API_KEY__`, optionally replaced by a CI secret) → `localStorage['simUni_apiKey']` → manual entry. There is a fully manual fallback ("Copiar prompt" + paste JSON) when no key/proxy is available.

**localStorage keys**: `simUni_apiKey`, `simUni_fetchedRes`.

**Automatic result updates** (added separately from the jsx): `scripts/fetch_results.mjs` runs in a scheduled GitHub Action (`.github/workflows/update-results.yml`, cron Mon/Thu + manual dispatch) on the GitHub repo `agamemnon140/brasileiro-2026`. It calls the Anthropic API with the `web_search_20260209` tool to find finished matches and writes `results.json` at the repo root, committing on change so GitHub Pages redeploys. A second weekly Action (`.github/workflows/update-tables.yml`, cron Fri + manual dispatch) runs `scripts/fetch_tables.mjs`: it discovers the current official CBF "Tabela Detalhada" PDFs via the public Strapi CMS (`cms.cbf.com.br/api/championship-documents`, type "Complemento de Tabela" — the blob URL hash changes weekly), downloads them, extracts played league-phase results with the Anthropic API (PDF document block) and merges into the same `results.json` (league phase A/B/C only; knockout data for Série D/Copa BR lives hardcoded in the app and is updated by hand via versioned releases). It skips games already hardcoded in the app by parsing `SA/SB/SC_RES` out of `index.html`. The app loads `results.json` on mount via a `useEffect` (injected just after `handleSearchResults` in `SimuladorUnificado`, in both `index.html` and `brasileirao-2026.html`) and feeds it through the same `handleSearchResults` pipeline. Server-side guardrails: canonical-team allowlist + `normName` map (kept in sync by hand with the jsx team lists), score range 0–14, required `source` URL, never overwrites a recorded score (logs conflicts), per-run volume cap. The Action needs the repo secret `ANTHROPIC_API_KEY`; the API key is never exposed client-side here.

## Conventions

- The big comment block at the top of the jsx is a **running changelog** (v4.x). When making a substantive change, add a dated/versioned entry there describing the fix and its rationale — this is the project's primary record of why things are the way they are.
- Series-format rules (relegation counts, access slots, playoff/quadrangular structures) reflect specific **2026 CBF regulations** and have been corrected repeatedly; verify against the regulation before changing zone cutoffs or `nReb`.
- Watch for call-stack overflow: the code uses a `minMax()` loop instead of `Math.min/max(...arr)` because Série D produces very large sample arrays (~960k).
